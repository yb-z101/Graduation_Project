# 基于大模型与工作流的对话式数据分析系统设计与实现

---

## 摘要

随着大数据时代的到来，企业和个人面临着日益增长的数据分析需求。传统的数据分析工具通常要求用户具备专业的SQL编程技能或熟练掌握复杂的数据处理软件，这使得数据分析的门槛居高不下。大语言模型（Large Language Model, LLM）的快速发展为自然语言交互式数据分析提供了新的技术路径，但现有系统仍存在意图理解不精准、代码执行安全性不足、多数据源支持有限、可视化与报告能力薄弱等问题。

本文设计并实现了一个基于大模型与工作流的对话式数据分析系统——"智析"，用户只需通过自然语言描述分析需求，系统即可自动完成意图识别、代码生成、数据查询、可视化展示和智能洞察等全流程操作。系统采用前后端分离的B/S架构，前端基于Vue 3框架，结合Element Plus组件库、ECharts和AntV G2双可视化引擎，构建了交互友好的BI工作台；后端基于FastAPI框架，集成LangGraph工作流引擎，实现了意图分类、Pandas代码生成与安全执行、SQL智能生成与校验等核心功能。系统支持CSV、Excel、SQL三种文件格式的数据上传，以及MySQL、PostgreSQL等数据库的直连分析，并提供了PDF报告导出、智能洞察生成、操作审计日志等企业级功能。

测试结果表明，系统能够准确识别用户的六类分析意图，有效生成并安全执行数据分析代码，支持多种可视化图表类型的自动生成，在典型数据分析场景下具有良好的可用性和实用性。

**关键词**：大语言模型；工作流；对话式数据分析；自然语言处理；数据可视化；LangGraph

## Abstract

With the advent of the big data era, enterprises and individuals face growing demands for data analysis. Traditional data analysis tools typically require users to possess professional SQL programming skills or proficiency in complex data processing software, keeping the barrier to data analysis high. The rapid development of Large Language Models (LLMs) provides a new technical approach for natural language interactive data analysis, but existing systems still suffer from imprecise intent understanding, insufficient code execution security, limited multi-data-source support, and weak visualization and reporting capabilities.

This paper designs and implements a conversational data analysis system based on LLMs and workflows called "ZhiXi", where users only need to describe their analysis needs in natural language, and the system automatically completes the entire process including intent recognition, code generation, data querying, visualization, and intelligent insights. The system adopts a B/S architecture with frontend-backend separation. The frontend is built on the Vue 3 framework, combined with the Element Plus component library and dual visualization engines of ECharts and AntV G2, constructing an interactive BI workbench. The backend is based on the FastAPI framework, integrating the LangGraph workflow engine to implement core functions such as intent classification, Pandas code generation and safe execution, and intelligent SQL generation with validation. The system supports data upload in CSV, Excel, and SQL formats, direct database connection analysis for MySQL and PostgreSQL, and provides enterprise-level features including PDF report export, intelligent insight generation, and operation audit logging.

Testing results demonstrate that the system can accurately recognize six types of analysis intents, effectively generate and safely execute data analysis code, support automatic generation of multiple visualization chart types, and exhibits good usability and practicality in typical data analysis scenarios.

**Keywords**: Large Language Model; Workflow; Conversational Data Analysis; Natural Language Processing; Data Visualization; LangGraph

## 第1章 绪论

### 1.1 研究背景

在数字经济快速发展的今天，数据已成为驱动企业决策和创新的核心资产。据IDC预测，全球数据总量将在2025年达到175ZB，企业对数据分析的需求呈指数级增长。然而，传统的数据分析流程存在显著的效率瓶颈：业务人员需要将分析需求转化为技术语言，数据分析师需要编写SQL查询或Python代码，整个流程耗时长、沟通成本高、迭代效率低。

传统的BI（Business Intelligence）工具如Tableau、Power BI虽然提供了可视化的拖拽式操作界面，但仍然要求用户理解数据模型和指标体系，学习成本较高。同时，这些工具在处理自然语言形式的开放式分析需求时能力有限，难以实现真正的"对话式"分析体验。在工业产品层面，Chat2DB作为GitHub上最受欢迎的Text-to-SQL工具，支持多种数据库连接和大模型集成[18]；PandasAI以19.9k Star的热度成为数据分析领域的热门开源项目[19]。然而，这些工具多局限于单一任务场景，对分析全流程的统一建模尚不充分。

近年来，以GPT、Qwen、DeepSeek为代表的大语言模型在自然语言理解和代码生成方面取得了突破性进展。LLM具备将自然语言描述转化为结构化查询语言（SQL）或编程代码的能力，为构建对话式数据分析系统提供了技术基础。同时，工作流编排技术（如LangGraph）的发展，使得将大模型的语义理解能力与流程管控能力深度融合成为可能。通过将LLM与工作流引擎深度融合，可以显著降低数据分析的使用门槛，使非技术背景的用户也能高效地进行数据探索和决策分析。

基于上述背景，本文设计并实现了"智析"——一个基于大模型与工作流的对话式数据分析系统，旨在通过自然语言交互的方式，让用户以最自然的表达方式完成从数据导入到洞察获取的全流程分析操作，为非技术人员提供"说话即查数"的轻量化分析体验。

### 1.2 研究现状

对话式数据分析是将自然语言处理技术与数据分析相结合的交叉研究领域，其核心目标是让用户无需掌握专业编程技能，即可通过自然语言交互完成数据查询、分析和可视化。该领域的发展可追溯至早期的自然语言接口到数据库研究，早期方法主要基于规则和模板匹配，但受限于技术水平，适用范围较为有限。随着深度学习和大语言模型技术的爆发式发展，该领域在2022年以后迎来了快速发展期。刘知远等[7]指出，该领域正从传统规则化流程向智能协作演进，从单模态向多模态融合，并提出构建"通用数据分析智能体"的新范式。当前研究主要集中在对话式智能体架构、工作流与大模型融合、Text-to-SQL技术、多轮交互上下文管理四个核心方向。

#### 1.2.1 对话式智能体架构研究

在对话式数据分析领域，国际学术界已涌现出一批具有代表性的创新成果。Martins等人于2025年发表在《IEEE Access》的系统性综述，对对话式智能体在可视化分析中的应用进行了全面梳理，指出自2022年以来该领域研究显著加速，现有系统日益采用多智能体推理、上下文感知提示和代码生成能力，但也存在提示脆弱性、模型幻觉等局限[3]。

Al Awad等人开发的Talk2Data系统采用多模态对话代理架构，集成Whisper语音识别、Qwen代码生成模型与沙箱执行环境，通过一个轻量级路由器集中决策代码生成还是对话响应，并将所有提示建立在紧凑的数据集上下文中。研究在48项测试任务上进行了验证，结果表明该系统能够提供透明、可审计的分析，同时保持对非技术用户的可访问性[2]。Zhang等人提出了InsightLens交互式系统，基于LLM的多智能体框架自动提取、关联和组织见解，通过与十二位数据分析师进行的用户研究表明，该系统可显著减少用户的手动和认知工作量[8]。

国内方面，刘知远等[7]从通用数据分析智能体的视角，系统性综述了大语言模型与智能体在数据分析领域的整体演进，指出语义理解、自主管道、自动工作流、工具协作与面向开放世界是五大核心方向。

#### 1.2.2 工作流与大模型融合研究

针对多轮对话中的状态管理问题，Kumar等人在NeurIPS 2025上提出的SEA（Stateful Execution Environment）框架，将数据分析工作流表示为有向无环图（DAG），通过双表示状态模型解耦轻量级符号状态图与完整计算状态图，实现了多轮对话中的状态复用，端到端任务成功率提升至84%[1]。这一思想为本课题的工作流编排设计提供了重要参考。

国内学者谢宇鹏等[9]提出了基于自然语言交互的数据分析系统Navi，采用模块化的设计原则，抽象出数据查询、可视化生成和可视化探索3个核心功能模块，并通过任务调度器实现各功能模块的有效协同。阿里云开发者社区[6]从产业实践的角度，总结了工作流与大模型的系统级融合趋势，指出嵌入式智能与流程级重构是两大特征，状态保持、领域知识动态注入与跨系统调用构成三大技术支柱。

#### 1.2.3 Text-to-SQL技术研究

Amazon Science团队[12]开发了SQLGenie系统，由表加载器、SQL生成器和反馈增强组件三个核心组件构成，在公开基准上达到了最先进的性能。CoQ框架团队[13]针对复杂SQL生成难度高的问题，提出了"逐句生成SQL，错了能回滚"的策略，在5大数据集上准确率超SOTA 13%+。Urban等人[4]聚焦于利用用户历史反馈优化Text-to-SQL翻译，该方法可使准确率最高提升14.9%。

国内方面，赵猛等[10]针对复杂自然语言查询转SQL的问题，提出了基于树状模型的技术，有效提高了SQL生成的准确性。陈洋、陈佳威[5]通过微调本地大模型实现自然语言交互的数据分析，并创新性地融入基于属性的访问控制策略，在确保数据安全的同时提升了分析准确性。

#### 1.2.4 多轮交互上下文管理研究

Li等[14]聚焦于多会话个性化对话问题，提出了大规模数据集CMT-Eval和分层树框架用于隐式推理，该数据集由596个高质量对话和4431轮对话组成。国内方面，刘超等[11]针对现有会话推荐系统存在的会话间信息挖掘不够充分的问题，提出了融合多层图与分类信息的双意图会话推荐模型，在五个公开数据集上的实验结果表明，该模型的P@20提升了1.23%~51.78%。思迈特Smartbi[31]提出了大模型驱动的数据分析新范式，指出实现可靠"意图识别"分析，需构建"指标层-语义层-RAG知识库-工作流"的四层融合架构。

#### 1.2.5 现有系统的不足

综合国内外研究可见，对话式数据分析正从单一模型向多智能体协作演进，从静态流程向动态工作流编排演进，从通用分析向垂直领域深度适配拓展。然而，现有研究和产品仍存在以下可探索空间：

（1）**意图理解不精准**：系统倾向于将所有数值列纳入分析，而非聚焦于用户关心的特定指标，导致分析结果冗余。

（2）**代码执行安全性不足**：LLM生成的代码可能包含危险操作，缺乏有效的沙箱隔离机制。

（3）**多数据源支持有限**：多数系统仅支持单一数据输入方式，难以同时覆盖文件上传和数据库连接两种场景。

（4）**可视化与报告能力薄弱**：图表生成依赖用户明确指定，缺乏智能推荐；报告内容单一，缺乏深度洞察。

（5）**大模型与工作流引擎的融合多停留在浅层集成**：缺乏对任务可观测性和可干预性的系统设计。

本课题拟在上述研究基础上，探索两者的深度融合路径，构建面向非技术用户的低门槛、高效率对话式数据分析系统。

### 1.3 研究内容

本文针对上述不足，设计并实现一个功能完善的基于大模型与工作流的对话式数据分析系统，主要研究内容包括：

（1）**多意图分类与精准响应机制**：设计基于关键词和正则表达式的六类意图分类器，结合上下文感知的数据裁剪策略和意图特定的Prompt模板，实现对用户分析意图的精准识别和差异化响应。

（2）**双路径分析引擎**：设计文件上传路径（Pandas代码生成与安全执行）和数据库连接路径（SQL生成与安全校验）的双路径分析架构，统一入口、差异化处理，覆盖数据分析的两种主要数据输入场景。

（3）**基于LangGraph的工作流编排**：借鉴SEA框架的DAG建模思想，使用LangGraph构建数据分析工作流，定义process_query、generate_visualization、clean_data三个核心节点，实现灵活的分析流程编排和状态管理。

（4）**多层次安全体系**：实现Pandas代码沙箱执行器（白名单函数+危险操作拦截）和SQL安全校验器（禁止DDL/DML危险语句），结合bcrypt密码加密和软删除机制，形成从代码执行到数据存储的完整安全链路。

（5）**智能可视化与报告系统**：实现基于查询内容的图表列筛选机制，支持柱状图、折线图、饼图、散点图、雷达图五种ECharts图表类型的自动生成；设计包含数据统计摘要、SQL语句、智能洞察等丰富内容的分析报告。

（6）**多LLM适配架构**：统一封装阿里云通义千问、DeepSeek、火山引擎豆包、讯飞星火四种大模型接口，支持运行时切换，提高系统的灵活性和可用性。

## 第2章 相关技术介绍

### 2.1 大语言模型

大语言模型（Large Language Model, LLM）是基于Transformer架构的大规模预训练语言模型，通过在海量文本数据上进行自监督学习，获得了强大的自然语言理解和生成能力。LLM的核心技术基础是2017年Google提出的Transformer架构，其自注意力机制（Self-Attention）使得模型能够捕捉文本中的长距离依赖关系。2020年，Brown等人提出的GPT-3模型[1]展示了大语言模型在少样本学习方面的惊人能力，标志着大语言模型时代的到来。

本文系统集成了以下四种大语言模型：

**（1）阿里云通义千问（Qwen）**

通义千问是阿里云推出的大语言模型系列，支持中英双语理解和生成。Bai等人[3]在Qwen技术报告中详细介绍了其模型架构和训练方法。本系统使用qwen-turbo模型，通过DashScope API调用，具备较快的响应速度和良好的中文理解能力，作为系统的默认模型。

**（2）DeepSeek**

DeepSeek是深度求索公司推出的大语言模型，在代码生成和数学推理方面表现突出。系统通过OpenAI兼容接口调用deepseek-chat模型，适用于需要复杂代码生成的分析场景。

**（3）火山引擎豆包（Doubao）**

豆包是字节跳动通过火山引擎平台提供的大语言模型服务。系统使用doubao-seed模型，作为备选模型提供冗余保障。

**（4）讯飞星火（Spark）**

星火是科大讯飞推出的大语言模型，在中文语义理解方面具有优势。系统通过其开放API接口调用spark-x模型。

系统设计了统一的LLM调用客户端，封装了模型选择、API调用、错误处理等通用逻辑，上层业务代码通过统一的`call_llm(model_id, prompt)`接口调用不同模型，实现了模型层面的解耦和可替换性。

### 2.2 LangGraph工作流引擎

LangGraph是LangChain生态系统中的工作流编排框架，基于有向图（Directed Graph）模型构建多步骤的AI应用。Kumar等人[1]提出的SEA框架将数据分析工作流表示为有向无环图（DAG），通过双表示状态模型解耦轻量级符号状态图与完整计算状态图，为工作流编排提供了重要的理论参考。LangGraph的核心概念包括：

**（1）状态（State）**：工作流中各节点共享的数据字典，节点通过读取和修改状态实现数据传递。这种设计借鉴了SEA框架中状态复用的思想，使得多轮对话中的中间结果可以被后续节点直接访问。

**（2）节点（Node）**：工作流中的处理单元，每个节点是一个Python函数，接收当前状态作为输入，返回状态更新。本系统定义了process_query（查询处理）、generate_visualization（可视化生成）和clean_data（数据清洗）三个核心节点。

**（3）边（Edge）**：定义节点之间的执行顺序和条件分支，支持无条件边和条件边两种类型。无条件边定义固定的执行顺序，条件边根据状态中的数据动态选择执行路径。

本系统使用LangGraph构建数据分析工作流，工作流根据用户意图和数据分析结果动态选择执行路径，实现了灵活的分析流程编排。相比传统的硬编码流程，LangGraph的工作流编排方式具有更好的可扩展性和可维护性，新增分析节点只需定义节点函数和添加边即可。

### 2.3 Vue 3前端框架

Vue 3是Vue.js框架的最新主版本，引入了Composition API、响应式系统重构和性能优化等重要改进。本系统前端采用Vue 3作为核心框架，主要使用以下特性：

**（1）Composition API**：使用`ref`、`computed`、`watch`等组合式API组织组件逻辑，相比Options API具有更好的代码组织和逻辑复用能力。例如，BI工作台组件通过Composition API将消息发送、图表管理、报告导出等逻辑组织为独立函数，提高了代码的可读性和可维护性。

**（2）Pinia状态管理**：使用Pinia替代Vuex进行全局状态管理，Pinia提供了更简洁的API和更好的TypeScript支持。系统定义了sessionStore（会话状态）、chatStore（聊天状态）和userStore（用户状态）三个Store，分别管理会话列表与当前会话、聊天消息与加载状态、用户信息与模型选择等状态。

**（3）Element Plus组件库**：基于Vue 3的企业级UI组件库，提供了丰富的表单、表格、对话框、消息提示等组件，显著提升了界面开发效率。系统使用Element Plus的el-container布局组件构建BI工作台的三栏布局，使用el-select实现模型切换，使用el-button实现功能操作按钮。

### 2.4 FastAPI后端框架

FastAPI是Python生态中高性能的Web框架，基于Starlette和Pydantic构建，具有以下特点：

**（1）异步支持**：原生支持async/await语法，适合处理I/O密集型的LLM API调用。系统中的文件上传、消息处理等接口均采用异步实现，提高了并发处理能力。

**（2）自动文档生成**：基于OpenAPI规范自动生成交互式API文档（Swagger UI），便于接口调试和协作开发。

**（3）类型验证**：利用Pydantic模型实现请求参数的自动验证和序列化，减少手动校验代码。系统定义了DatasourceCreateRequest、TaskCreateRequest等Pydantic模型，确保接口参数的类型安全。

**（4）依赖注入**：通过`Depends`机制实现数据库会话、认证信息等依赖的自动注入，简化了代码结构。系统通过`get_db`依赖注入函数自动管理数据库会话的创建和关闭。

本系统后端使用FastAPI构建RESTful API，通过SQLAlchemy ORM操作MySQL数据库，通过CORS中间件支持前端跨域请求，通过全局异常处理器统一处理错误响应。

### 2.5 数据可视化技术

本系统采用双可视化引擎架构：

**（1）ECharts**：百度开源的JavaScript可视化图表库，支持柱状图、折线图、饼图、散点图、雷达图等丰富的图表类型，配置灵活、交互性强。系统后端通过chart_generator自动生成ECharts配置，前端通过DataChart组件渲染。ECharts的配置采用JSON格式，后端生成的配置可直接传递给前端使用，实现了前后端的解耦。

**（2）AntV G2**：蚂蚁集团推出的可视化语法引擎，基于图形语法（Grammar of Graphics）理论，擅长处理统计图表和复杂可视化场景。系统前端在DataChart组件中集成了G2引擎，作为ECharts的补充，用于处理ECharts不擅长的高级可视化需求。

双引擎架构的设计使得系统既能利用ECharts丰富的图表类型和成熟的生态，又能在需要时切换到G2引擎实现更复杂的可视化效果，提高了系统的可视化能力上限。

### 2.6 本章小结

本章介绍了系统开发中涉及的关键技术。大语言模型为系统提供了自然语言理解和代码生成的核心能力；LangGraph工作流引擎为系统提供了灵活的分析流程编排和状态管理能力；Vue 3和FastAPI分别作为前后端框架，为系统提供了高性能的Web应用开发基础；ECharts和AntV G2双可视化引擎为系统提供了丰富的数据可视化能力。这些技术的有机结合，为对话式数据分析系统的设计与实现提供了坚实的技术基础。

## 第3章 系统需求分析

### 3.1 功能需求分析

#### 3.1.1 数据导入功能

系统需要支持多种数据源的导入，满足不同用户的数据分析需求：

**FR-01 文件上传**：支持CSV、Excel（.xlsx）、SQL三种格式的文件上传。CSV和Excel文件通过Pandas解析为DataFrame，SQL文件通过SQLite内存数据库执行后提取表数据。对于Excel文件，需支持多Sheet解析；对于SQL文件，需支持多表数据提取，并自动进行MySQL到SQLite的语法转换。

**FR-02 数据库连接**：支持MySQL和PostgreSQL数据库的直连分析。用户通过表单输入主机、端口、用户名、密码、数据库名等信息建立连接，密码需通过bcrypt加密存储。连接建立后可浏览数据库表结构，执行SQL查询。

**FR-03 文件预览**：上传文件后展示数据预览（前5行），包括列名、数据类型、行数等基本信息，帮助用户了解数据概况。对于SQL文件，还需展示包含的表数量和各表的行数。

#### 3.1.2 对话分析功能

**FR-04 自然语言查询**：用户通过自然语言描述分析需求，系统自动识别意图并生成对应的分析代码或SQL语句。支持的查询类型包括：数据展示、排名排序、统计分析、条件查询、趋势对比、可视化图表等。

**FR-05 意图识别**：系统自动识别用户的六类分析意图（数据展示、排名排序、可视化、统计分析、条件查询、趋势对比），并根据意图类型选择差异化的处理策略和Prompt模板。

**FR-06 代码生成与执行**：对于文件上传模式，系统生成Pandas代码并在沙箱环境中安全执行；对于数据库连接模式，系统生成SQL语句并经安全校验后执行。

**FR-07 多表分析**：对于SQL文件中的多表数据，系统支持按表名查询特定表，也支持跨表关联分析。采用三层架构（意图分类→上下文裁剪→精准Prompt）处理多表查询，分析结果按表名去重展示，避免数据冗余。

**FR-08 多轮对话上下文记忆**：系统支持至少3轮的多轮对话上下文记忆，在生成代码或SQL时结合历史对话内容，实现连续分析和钻取查询。

#### 3.1.3 数据可视化功能

**FR-09 图表自动生成**：根据用户查询内容自动生成ECharts图表配置，支持柱状图、折线图、饼图、散点图、雷达图五种图表类型。图表仅展示用户查询中提到的数值列，避免无关数据的干扰。

**FR-10 图表交互**：图表支持缩放、筛选、数据点提示等交互操作。用户可将图表钉住到看板页面，进行持久化展示和对比分析。

**FR-11 图表导出**：支持将图表导出为PNG图片，或嵌入PDF/HTML分析报告中。

#### 3.1.4 报告导出功能

**FR-12 HTML报告导出**：生成包含会话信息、数据统计摘要、智能洞察、SQL语句、对话记录、数据预览（去重）、交互式图表等内容的HTML格式分析报告。

**FR-13 PDF报告导出**：基于ReportLab生成PDF格式分析报告，支持图表截图嵌入、Markdown格式化、中文字体渲染。

**FR-14 智能洞察**：基于用户查询问题、数据统计摘要和对话记录，调用LLM生成2-3条具有决策参考价值的智能洞察，嵌入分析报告中。

#### 3.1.5 系统管理功能

**FR-15 会话管理**：支持新建会话、切换会话、删除会话、清空全部会话。会话状态通过Pinia Store和localStorage持久化。

**FR-16 执行日志**：展示分析流程的执行过程，包括意图识别、SQL生成、数据分析、图表生成等步骤的状态和时间线。

**FR-17 审计日志**：记录用户的所有操作行为，包括查询内容、操作类型、执行状态、时间戳等，支持分页查询、多条件筛选和导出。

**FR-18 数据清洗**：支持通过自然语言描述清洗需求，系统生成Pandas清洗代码并执行，如处理缺失值、去除重复行、异常值修正等。

#### 3.1.6 模型管理功能

**FR-19 多模型切换**：支持在阿里云Qwen、DeepSeek、火山引擎豆包、讯飞星火四种模型之间切换，适应不同场景的分析需求。

### 3.2 非功能需求分析

**NFR-01 性能要求**：系统响应时间应在5秒以内（不含LLM调用时间），LLM调用的超时时间设置为60秒。支持同时处理多个会话的并发请求。

**NFR-02 安全要求**：Pandas代码执行必须在沙箱环境中进行，禁止访问文件系统、网络和系统命令；SQL语句必须经过安全校验，禁止执行DDL和DML操作；数据源密码必须通过bcrypt加密存储。

**NFR-03 可用性要求**：系统界面应简洁直观，非技术背景用户无需培训即可使用。自然语言查询的容错率应较高，能够理解模糊或不完整的表述。

**NFR-04 可扩展性要求**：系统应支持新增数据源类型、图表类型和LLM模型，模块间通过接口解耦，便于功能扩展。LLM调用客户端采用统一接口设计，新增模型只需实现对应的调用函数即可。

**NFR-05 兼容性要求**：前端应兼容主流浏览器（Chrome、Firefox、Edge、Safari），后端应兼容MySQL和PostgreSQL数据库。

### 3.3 本章小结

本章从功能需求和非功能需求两个维度对系统进行了详细分析。功能需求方面，系统需要支持CSV、Excel、SQL三种格式数据上传和数据库直连，实现六类意图识别与差异化响应，支持柱状图、折线图、饼图、散点图、雷达图五种图表类型，提供HTML和PDF两种报告导出格式，以及会话管理、审计日志、数据清洗等系统管理功能。非功能需求方面，系统需要在性能、安全、可用性、可扩展性和兼容性等方面达到预期指标。这些需求为后续的系统设计和实现提供了明确的指导。

#### 4.2.2 对话分析模块

图4-3 对话分析模块处理流程

**意图识别**阶段，系统通过IntentClassifier对用户查询进行分类，支持六种意图类型：数据展示（data_display）、排名排序（ranking_sort）、可视化（visualization）、统计分析（statistical_analysis）、条件查询（conditional_query）和趋势对比（trend_comparison）。分类结果决定了后续的Prompt模板选择和响应策略。

**代码/SQL生成**阶段，系统根据数据源类型选择不同的生成策略：文件上传模式调用`generate_pandas_code`生成Pandas代码，数据库连接模式调用`generate_sql`生成SQL语句。对于SQL文件中的多表场景，系统还实现了上下文感知的数据裁剪策略（build_context_aware_data），根据意图类型只提取相关的数据摘要，避免信息过载导致LLM生成冗余回复。

**安全执行**阶段，Pandas代码在safe_executor沙箱中执行，白名单内置函数，禁止import、open、eval、exec、os、subprocess等危险操作；SQL语句通过security.py校验，仅允许SELECT和WITH开头的查询语句。

**分析摘要生成**阶段，系统对硬编码逻辑无法覆盖的查询场景，回退到LLM生成分析摘要，确保回答的准确性和完整性。

### 5.2 LLM统一调用客户端实现

系统通过llm_client.py实现了统一的LLM调用客户端，封装了四种大模型的API调用逻辑。核心实现包括模型配置、统一调用接口和专用生成函数三个部分。

系统定义了四个模型配置，每个配置包含API地址、密钥、模型名称和请求参数。配置信息通过Pydantic Settings从环境变量中读取，确保敏感信息不硬编码在代码中。`call_llm`函数根据model_id选择对应的配置，构建请求体，通过HTTP POST调用模型API，解析响应并返回生成的文本。函数还包含了错误处理和重试逻辑，当API调用失败时自动重试最多3次，每次间隔2秒。

四种模型的API调用方式各有差异：Qwen使用DashScope API的input/parameters格式，DeepSeek和火山引擎使用OpenAI兼容的messages格式，星火使用HTTP API Password认证方式。统一调用客户端屏蔽了这些差异，上层业务代码只需指定model_id即可调用任意模型。

此外，系统还实现了`generate_pandas_code`、`generate_sql`、`generate_clean_code`等专用函数，通过精心设计的Prompt模板引导LLM生成特定格式的代码输出，并从响应中提取代码块。`generate_pandas_code`函数在Prompt中明确要求代码将最终结果赋值给变量`result`，且仅使用pandas和Python内置函数；`generate_sql`函数要求仅返回可执行的SELECT查询语句，禁止包含DROP/DELETE/ALTER等危险操作。

### 5.3 意图分类器实现

意图分类器（IntentClassifier）采用关键词匹配和正则表达式相结合的方式进行意图识别。系统定义了六种意图模式，每种模式包含关键词列表、正则模式、优先级和响应风格，如表5-2所示。

表5-2 意图分类模式定义

| 意图类型 | 关键词示例 | 正则模式示例 | 优先级 | 响应风格 |
|---------|-----------|-------------|-------|---------|
| 数据展示 | 展示、显示、查看、所有 | 列出.*(?:所有\|全部) | 1 | summary_table |
| 排名排序 | 排名、排序、top、前几 | 按.*排(?:名\|序) | 2 | ranked_list |
| 可视化 | 图表、画图、柱状图、折线图 | 无 | 3 | chart_focused |
| 统计分析 | 平均、均值、总和、最大值 | 最(?:大\|小)(?:的)? | 4 | statistical_summary |
| 条件查询 | 多少个、几个、筛选、条件 | .*的.*是(?:多少\|哪些) | 5 | filtered_result |
| 趋势对比 | 趋势、变化、增长、对比 | .*(?:趋势\|变化\|增长) | 6 | trend_analysis |

分类过程分两步：首先遍历所有意图的正则模式进行匹配，匹配成功则直接返回；若正则未匹配，则统计每种意图的关键词命中数，按命中数降序、优先级升序排序，返回最佳匹配。若所有模式均未匹配，则归为通用查询（general_query）。

此外，IntentClassifier还提供了`extract_chart_intent`方法，专门识别用户的图表类型偏好，支持柱状图、折线图、饼图、散点图和雷达图五种类型的识别。该方法与意图分类器配合使用，当用户表达可视化需求时，同时确定具体的图表类型。

### 5.4 LangGraph工作流实现

系统使用LangGraph的StateGraph构建数据分析工作流。借鉴SEA框架[1]的DAG建模思想，工作流定义了三个节点：

**process_query节点**：核心处理节点，根据数据源类型（文件上传/SQL文件/数据库连接）选择不同的处理路径。对于文件上传模式，生成Pandas代码并执行；对于SQL文件，采用三层架构（意图分类→上下文裁剪→精准Prompt）处理多表查询；对于数据库连接，生成SQL并执行。处理完成后生成分析摘要和图表配置。

process_query节点中，上下文感知的数据裁剪策略（build_context_aware_data）是关键设计。该函数根据意图类型提取不同粒度的数据摘要：排名排序意图提取Top5排名数据，可视化意图提取数值列的min/max/avg统计，统计分析意图提取完整的describe统计摘要，其他意图提取前3行数据样本。这种差异化裁剪策略有效避免了信息过载导致LLM生成冗余回复的问题。

**generate_visualization节点**：可视化生成节点，仅在用户明确请求生成图表时才调用chart_generator根据分析结果和用户查询生成ECharts图表配置。

**clean_data节点**：数据清洗节点，根据用户的清洗指令生成Pandas清洗代码并执行。

工作流的边定义了节点间的执行顺序：process_query完成后进入generate_visualization，clean_data作为可选分支。工作流通过`create_analysis_workflow`函数创建并编译，返回可执行的工作流实例。

### 5.5 安全执行器实现

Pandas代码安全执行器（safe_executor.py）是系统安全体系的核心组件。执行器通过以下机制确保代码安全：

**（1）白名单函数**：只允许使用Pandas、NumPy等数据处理相关的内置函数，如len、range、sum、max、min、abs、round、sorted、enumerate、zip等，共计20余个安全函数。

**（2）危险操作拦截**：禁止使用import、open、eval、exec、os、subprocess、sys等危险操作，通过13个正则表达式模式检测代码中的危险关键词。

**（3）沙箱环境**：代码在受限的全局命名空间中执行，只暴露df（DataFrame变量）、pd（Pandas模块）、np（NumPy模块）和白名单函数，无法访问外部系统资源。代码中的import语句会被自动移除，因为必要的模块已在沙箱环境中预置。

SQL安全校验器（security.py）通过正则表达式检测SQL语句中的危险操作，仅允许SELECT和WITH开头的查询语句，禁止DROP、DELETE、ALTER、TRUNCATE、INSERT、UPDATE、CREATE、RENAME等DDL和DML操作。同时校验括号匹配，防止SQL注入。

### 5.6 图表生成器实现

图表生成器（chart_generator.py）根据DataFrame和用户查询自动生成ECharts图表配置。生成过程包括三个步骤：

**步骤一：图表类型判定**。根据用户查询中的关键词确定图表类型，如"柱状图"→bar，"折线图"→line。若用户未指定图表类型，则不生成图表（返回空配置），避免无关可视化。

**步骤二：相关列筛选**。`_filter_relevant_cols`函数从用户查询中匹配数值列名。例如用户问"用柱状图展示库存量"时，只匹配"库存量"列。当查询中未匹配到任何列名时，回退到展示全部数值列。

**步骤三：配置生成**。根据图表类型调用对应的生成函数，每种函数生成符合ECharts规范的option配置对象。生成过程中自动识别分类列和数值列，处理日期列，计算统计值，生成标题和提示框配置。饼图生成时智能选择分类列，优先选择性别相关列，避免使用姓名列作为分类依据。

### 5.7 报告导出与智能洞察实现

HTML报告导出在前端实现，通过构建完整的HTML字符串并触发浏览器下载。报告包含会话信息、数据统计摘要、智能洞察、SQL语句、对话记录、数据预览和交互式图表等板块。

数据预览采用去重策略：使用Set记录已出现的表名，同一张表只出现一次，展示前5行数据。图表通过ECharts CDN在HTML报告中交互式渲染，每个图表使用独立的DOM容器和JavaScript初始化代码。

PDF报告导出在后端实现，基于ReportLab库生成。报告包含数据概览、对话记录、分析总结、图表截图等板块。Markdown格式的文本通过`_format_markdown_for_pdf`方法转换为ReportLab支持的HTML标签，使用正则表达式处理粗体和代码块。图表截图以Base64 PNG格式通过API传递，使用ReportLab的ImageReader解码并嵌入PDF。

智能洞察功能通过后端API实现，收集用户查询问题、数据统计摘要和对话记录摘要作为LLM的输入。LLM的Prompt要求生成2-3条关键洞察，每条洞察必须引用具体的数据和数值，具有决策参考价值。

### 5.8 前端核心组件实现

#### 5.8.1 BI工作台（BIWorkbench.vue）

BI工作台是系统的主页面，采用三栏布局：左侧为数据资源树（DataResourceTree），中间为对话面板（ChatPanel），右侧为分析结果工作区（WorkspacePanel），底部为执行过程面板（ExecutionProcessPanel）。

工作台的核心逻辑是`handleSendMessage`函数，负责发送用户消息、接收后端响应、更新UI状态。函数实现了以下关键逻辑：

**（1）统一去重策略**：通过`findExistingTab`和`upsertTab`函数，对分析结果Tab进行去重。同表名的Tab更新数据而非重复创建，图表Tab使用基于查询内容的独特标题（如"图表-用柱状图展示库存量"），确保多个图表可以同时保留。

**（2）多类型响应处理**：根据后端响应中的`is_multi_table_response`、`result`、`chart_option`等字段，分别处理多表响应、单表响应和图表响应。

**（3）异步报告生成**：`handleExportReport`函数改为async，先调用智能洞察API获取洞察内容，再生成HTML报告。

#### 5.8.2 数据图表组件（DataChart.vue）

DataChart组件支持ECharts和AntV G2双引擎渲染。组件接收chartOption属性，自动检测配置格式选择对应的渲染引擎。组件还提供了`getChartImageBase64`方法，通过ECharts的`getDataURL`API获取图表的Base64截图，用于报告导出。

组件支持钉住到看板、复制图表配置、下载PNG图片等操作。钉住功能通过localStorage持久化图表数据，在看板页面恢复展示，实现了图表的跨页面共享。

#### 5.8.3 会话状态管理

前端使用Pinia定义了三个Store：sessionStore管理会话列表与当前会话ID，chatStore管理聊天消息与加载状态，userStore管理用户信息与模型选择。会话状态同时持久化到localStorage，确保页面刷新后会话信息不丢失。

### 5.9 本章小结

本章详细描述了系统各功能模块的具体实现过程。LLM统一调用客户端封装了四种大模型的API调用逻辑，通过统一接口实现模型层面的解耦；意图分类器采用关键词和正则相结合的方式实现六类意图识别；LangGraph工作流编排了三个核心节点，实现了状态驱动的分析流程；安全执行器通过白名单函数、危险操作拦截和沙箱环境三层机制确保代码执行安全；图表生成器实现了基于查询内容的相关列筛选和五种图表类型的自动生成；报告导出支持HTML和PDF两种格式，智能洞察功能基于LLM生成决策参考建议；前端BI工作台采用三栏布局，实现了统一去重、多类型响应处理等关键逻辑。这些实现共同构成了功能完善的对话式数据分析系统。

## 第7章 总结与展望

### 7.1 工作总结

本文设计并实现了一个基于大模型与工作流的对话式数据分析系统"智析"，主要工作和贡献如下：

**（1）设计了六类意图分类与精准响应机制**。通过关键词和正则表达式相结合的意图分类器，实现了对用户分析意图的精准识别。针对不同意图类型设计了差异化的Prompt模板和响应策略，有效提升了LLM回复的针对性和质量。上下文感知的数据裁剪策略根据意图类型提取不同粒度的数据摘要，避免了信息过载导致的冗余回复。

**（2）实现了双路径分析引擎**。设计了文件上传路径（Pandas代码生成与安全执行）和数据库连接路径（SQL生成与安全校验）的双路径架构，统一入口、差异化处理，覆盖了数据分析的两种主要数据输入场景。对于SQL文件中的多表场景，采用三层架构（意图分类→上下文裁剪→精准Prompt）处理，实现了智能化的多表查询。

**（3）构建了基于LangGraph的工作流编排引擎**。借鉴SEA框架[1]的DAG建模思想，使用LangGraph构建数据分析工作流，定义了process_query、generate_visualization、clean_data三个核心节点，实现了状态驱动的分析流程编排和灵活的执行路径选择。

**（4）构建了多层次安全体系**。实现了Pandas代码沙箱执行器（白名单函数+危险操作拦截）和SQL安全校验器（禁止DDL/DML），结合bcrypt密码加密和软删除机制，形成了从代码执行到数据存储的完整安全链路。

**（5）设计了智能可视化与报告系统**。实现了基于查询内容的图表列筛选机制，解决了LLM倾向于展示所有数值列的问题；设计了包含数据统计摘要、SQL语句、智能洞察等丰富内容的分析报告，并实现了数据预览去重和多图表展示。

**（6）实现了多LLM适配架构**。统一封装了阿里云通义千问、DeepSeek、火山引擎豆包、讯飞星火四种大模型接口，支持运行时切换，提高了系统的灵活性和可用性。

### 7.2 不足与展望

尽管系统已实现了预期的核心功能，但仍存在以下不足和改进空间：

**（1）意图分类精度有限**。当前基于关键词和正则的意图分类方法在处理复杂或模糊的自然语言表述时可能分类不准确。未来可考虑使用LLM本身进行意图分类，或引入轻量级的文本分类模型提升分类精度。

**（2）会话状态存储在内存中**。当前会话数据（包括完整的DataFrame）存储在Python进程的内存字典中，服务重启即丢失，且大数据量时内存占用高。未来可引入Redis缓存或将会话数据持久化到数据库。

**（3）缺乏用户认证体系**。当前系统未实现用户登录和权限控制，所有用户共享同一套数据。未来可引入JWT认证和基于角色的访问控制（RBAC），满足多用户场景的需求。

**（4）图表类型覆盖有限**。当前仅支持五种基础图表类型，未来可扩展支持热力图、桑基图、地图等高级图表类型，并引入LLM自动推荐图表类型的能力。

**（5）缺乏数据导出功能**。当前只能导出分析报告，无法将分析结果或筛选后的数据导出为CSV/Excel格式。未来可增加数据导出功能，提升数据分析的闭环体验。

**（6）硬编码分析逻辑可优化**。部分分析摘要的生成仍依赖硬编码的关键词匹配逻辑，虽然已实现LLM回退机制，但硬编码分支仍可能产生不准确的回答。未来可进一步减少硬编码逻辑，将更多场景交由LLM处理。

**（7）RAG检索增强生成待完善**。系统已设计了RAG向量索引表，但检索增强生成功能尚未完整实现。未来可引入向量数据库（如Milvus、Chroma），实现基于历史SQL和业务文档的检索增强，提升SQL生成的准确性和领域适配能力。
